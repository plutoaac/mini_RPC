# 心跳机制设计文档

## 1. 设计概述

本文档描述了 RPC 框架中基于**复用现有协议层**实现的心跳机制。该方案通过保留服务名 `"__Heartbeat__"` 作为特殊标识，在不改动 Protobuf 定义、不增加底层心跳包的前提下，实现了连接活性探测。

### 核心设计原则

| 原则           | 说明                                                           |
| -------------- | -------------------------------------------------------------- |
| **协议复用**   | 心跳复用现有 `RpcRequest/RpcResponse` 帧格式，不引入新消息类型 |
| **非侵入式**   | 不修改 `.proto` 文件，不修改 Protobuf 生成代码                 |
| **快速路径**   | 服务端识别心跳后直接返回，不进线程池、不进业务分发             |
| **写缓冲安全** | 心跳响应复用 `EnqueueResponse` 路径，不破坏非阻塞写时序        |

## 3. 详细时序图

### 3.1 正常心跳流程（连接活跃）

```
Client                              Server
  │                                   │
  │  ── RpcRequest ──▶               │
  │     service="__Heartbeat__"       │
  │     method="Ping"                 │
  │     request_id="hb_123"           │
  │     payload=""                    │
  │                                   │ 识别 __Heartbeat__
  │                                   │ 快速路径：
  │                                   │ - 不进线程池
  │                                   │ - 不进服务注册表
  │                                   │ - 构造空 Response
  │                                   │
  │  ◀── RpcResponse ──              │
  │     request_id="hb_123"           │
  │     error_code=OK                 │
  │     payload=""                    │
  │                                   │
  │ 识别 hb_ 前缀 → 仅记录日志         │
  │ 更新 last_activity_time_          │
  │                                   │
  │  ── RpcRequest ──▶               │ 正常业务调用
  │     service="TestService"         │
  │     ...                           │
  │                                   │
```

### 3.2 心跳超时检测（连接失活）

```
Client                              Server
  │                                   │
  │ last_activity = T0                │
  │                                   │
  │ (等待 30s...)                     │ (网络故障/服务端宕机)
  │                                   │ 客户端无法收到任何数据
  │                                   │
  │ tick: idle = 32s > 30s            │
  │ → SendHeartbeat()                 │ (消息发出但无响应)
  │ last_activity = T0 (不变)         │
  │                                   │
  │ (等待 15s...)                     │
  │                                   │
  │ tick: idle = 47s > 45s            │
  │ → connection inactive!            │
  │ → FailAll("heartbeat timeout")    │
  │ → dispatcher_running = false      │
  │ → Close()                         │
  │                                   │
```

### 为什么不用 `write(fd)` 直接写？

```
✗ 错误方式 (同步写，会破坏非阻塞模型):
  write(fd, data, len)  → 可能阻塞当前线程
                        → 违反事件循环的非阻塞设计
                        → 可能触发 SIGPIPE

✓ 正确方式 (复用现有写路径):
  EnqueueResponse() → QueueResponse() → write_buffer_
                   → FlushWrites()    → 非阻塞 send()
                   → 等待 EPOLLOUT    → 继续发送
```

## 6. 心跳响应处理

### 为什么心跳响应不走 `pending_calls_->Complete()`？

```
正常 RPC 响应流程:
  request_id = "123"  →  pending table 中查找 "123"
                       → 找到 Slot → Complete() → 唤醒等待线程

心跳响应流程:
  request_id = "hb_123" → pending table 中查找 "hb_123"
                        → ❌ 找不到! (心跳请求未注册到 pending table)
                        → 如果不特判，会打印 "unknown response" 警告

解决方案:
  在 HandleReadableFrames() 中识别 "hb_" 前缀:
    if (request_id.rfind("hb_", 0) == 0) {
      // 心跳 ACK，仅记录日志，不走 Complete()
      LogInfo("heartbeat ack received");
    }
```

## 7. 可配置项

通过 `RpcClientOptions` 配置:

```cpp
RpcClientOptions options;
options.heartbeat_interval = std::chrono::seconds{30};  // 默认 30s
options.heartbeat_timeout  = std::chrono::seconds{45};  // 默认 45s

// 禁用心跳:
options.heartbeat_interval = std::chrono::seconds{0};
options.heartbeat_timeout  = std::chrono::seconds{0};
```

| 配置项               | 默认值 | 说明                               |
| -------------------- | ------ | ---------------------------------- |
| `heartbeat_interval` | 30s    | 超过此时间没收到数据，发送心跳探测 |
| `heartbeat_timeout`  | 45s    | 超过此时间没收到数据，关闭连接     |
| 设为 `0`             | —      | 禁用对应功能                       |

## 8. 关键设计决策

### 8.1 为什么不使用底层 TCP Keepalive？

| 特性         | TCP Keepalive        | 应用层心跳         |
| ------------ | -------------------- | ------------------ |
| 检测粒度     | 系统级，通常 2 小时+ | 应用级，可精细控制 |
| 跨平台一致性 | 不同 OS 行为差异大   | 行为完全一致       |
| 可观测性     | 内核态，难以监控     | 应用态，日志可追踪 |
| 与业务集成   | 独立于业务           | 可复用现有协议栈   |

### 8.2 为什么不修改 Protobuf 消息？

- **改动最小化**: 不需要重新生成 `.pb.cc/.pb.h`
- **向后兼容**: 旧版本服务端不认识 `__Heartbeat__` 会返回 `METHOD_NOT_FOUND`
- **面试友好**: 可以讲"复用应用层 RPC 协议实现 heartbeat fast path"

### 8.3 心跳 request_id 为什么用 `hb_` 前缀？

- 简单可靠的识别方式
- 无需修改 Protobuf 定义
- `rfind("hb_", 0)` 是 O(1) 的前缀检查

## 10. 风险与缓解措施

| 风险                                    | 缓解措施                                                             |
| --------------------------------------- | -------------------------------------------------------------------- |
| 心跳响应不走 `pending_calls_`，可能误报 | 通过 `hb_` 前缀特判，跳过 `Complete()`                               |
| 服务端直接 `write(fd)` 破坏非阻塞模型   | 复用 `EnqueueResponse` 写缓冲路径                                    |
| dispatcher 线程自 join                  | heartbeat timeout 只设置 `dispatcher_running_=false`，由循环自然退出 |
| 心跳风暴（网络恢复后连发）              | 独立维护 `last_heartbeat_time_` 记录发送时间，避免连续发送心跳包     |
| 频繁心跳影响性能                        | 默认 30s 间隔，对正常业务调用几乎无影响                              |

## 11. 已知问题与修复记录 (Bug Fix)

### 11.1 心跳超时条件永远无法触发的问题

**故障现象：**
最初版本中，为了防止“心跳风暴”（短时间内发送大量心跳请求），在发出心跳后直接将 `last_activity_time_` 更新成了当前时间 `now`。
由于默认的 `heartbeat_interval` 为 30s，而 `heartbeat_timeout` 为 45s。这导致闲置时间计算出现逻辑闭环：
- 等待 30s 时，发送心跳，并将 `last_activity_time_` 归零。
- 等待 45s 时，距离上次 `last_activity_time_` 仅过去了 15s，无法达到超时阈值。
- 等待 60s 时，再次发送心跳并将时间归零...
- **结果：** 客户端永远认为此时连接尚未超时，即使服务端实际上已经失联，也只会每 30s 徒劳地向虚空发送心跳包，永远无法触发断开（超时）。

**修复方案：**
通过拆分两个独立的时间戳以隔离不同的行为意图：
1. **`last_activity_time_`（连接活跃探针）：** 唯一职责是记录**最后一次成功收到服务端有效数据**的时间。它直接用于测算闲置时间，决定是否触发 45s 的心跳超时断开。
2. **`last_heartbeat_time_`（心跳发送节流流）：** 独立记录**最后一次主动发出心跳探测**的时间。当准备发送心跳前校验该值，既能保证正常发送，又能从根本上防止短时间产生心跳风暴。

通过这套双时间戳机制，我们兼顾了发送频率节流与真实断连检测，修复了连接无法正常释放的重大缺陷。
